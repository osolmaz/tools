# Daily Work Summary

Use this skill to answer questions like:

- `what did this person do in the last 24 hours?`
- `give me a daily summary for osolmaz/dutifulbob`
- `what the hell did this guy do bro?`
- `summarize their work yesterday across this org`

## Workflow

1. Define the scope before collecting data.

- Identify the time window using concrete timestamps.
- Identify whether multiple GitHub usernames are one person or separate people.
- Identify the orgs/repos to include.
- Respect explicit exclusions such as `no comments`, `only merged`, `only openclaw org repos`, or `include closed`.

2. Query GitHub with `gh`, not the GitHub connector.

- Prefer `gh search prs` and `gh search issues` for broad discovery.
- Search both `--author` and `--involves` when the user expects all activity, but do not automatically report old third-party items that only had label/comment churn.
- Use `--owner <org>` or repo qualifiers when the user gives an org/repo boundary.
- Fetch details with `gh pr view` or `gh issue view` when state, merge status, author, title, timestamps, labels, or URLs are unclear.

Useful starting commands:

```bash
gh search prs --owner ORG --author USER --updated ">=YYYY-MM-DDTHH:MM:SSZ" --json repository,number,title,state,isDraft,author,createdAt,updatedAt,closedAt,url
gh search prs --owner ORG --involves USER --updated ">=YYYY-MM-DDTHH:MM:SSZ" --json repository,number,title,state,isDraft,author,createdAt,updatedAt,closedAt,url
gh search issues --owner ORG --author USER --updated ">=YYYY-MM-DDTHH:MM:SSZ" --json repository,number,title,state,author,createdAt,updatedAt,closedAt,url
gh search prs automerge --owner ORG --commenter USER --match comments --updated ">=YYYY-MM-DDTHH:MM:SSZ" --json repository,number,title,state,author,updatedAt,url
```

3. Decide what counts as reportable work.

Include:

- PRs authored by the target identity in the window.
- PRs merged, closed, or materially updated in the window when they belong to the target identity.
- Replacement/follow-up PRs generated by automation when they clearly replace or land the target identity's work.
- Issues opened by the target identity when they are planning, prompt requests, triage requests, or task requests.
- Closed or merged work when the user asks for closed work or a full daily report.
- PRs by other authors that the target identity explicitly merged, approved, or routed into merge automation.
- Merge-command comments such as `@clawsweeper automerge`, `@clawsweeper approve`, `/merge`, or equivalent maintainer approval commands.

Exclude unless the user asks:

- Comments as standalone activity.
- Comments that only ask for review, explain status, add proof, or discuss implementation.
- Old third-party PRs/issues that only changed because of label churn, bot comments, or review routing.
- Draft-only bookkeeping that does not reflect a concrete work item.
- Per-user attribution when the user says multiple identities are one person.

4. Verify suspicious gaps.

- If the result feels too small, broaden from `--author` to `--involves`.
- Check both PRs and issues.
- Check closed/merged states, not only open work.
- If a bot-authored PR appears, inspect labels/title/timeline enough to decide whether it is a real continuation of the person's work or just automation noise.
- Check merge-command activity separately from authored work. Filter out PRs authored by the target identity when the user asks for work from others that the target merged.

5. Cluster repeated work into one item.

- If several PRs/issues are the same work item, report them as one bullet.
- Cluster closed/replaced PRs with the replacement PR that landed or remains open.
- Cluster bot follow-up PRs with the human-authored PR they replaced when the relationship is clear.
- Append all source links at the end of the same bullet, in the order that best explains the story.

## Output Format

Use concise sections by default:

- `Shipped`
- `Closed / Replaced`
- `In Progress`
- `Merged / Approved Others`
- `Planning`

Omit empty sections.

Each bullet must be plain language and append the link at the end:

```md
- Generic embedding-provider contract so plugins can provide embeddings through a shared API. https://github.com/org/repo/pull/123
```

When multiple PRs/issues are the same work item, keep one bullet and put multiple links at the end:

```md
- Provider-auth startup fix so gateway prewarm waits until after startup. https://github.com/org/repo/pull/456 https://github.com/org/repo/pull/457
```

## Style Rules

- Do not say `worked on`, `built a draft`, `started working on`, or other filler that adds no information.
- Prefer nouns and concrete outcomes: `OpenAI-compatible embeddings...`, `Provider-auth startup fix...`, `ClawSweeper prompt request...`
- Keep bullets short enough to scan.
- Do not quote comments or PR bodies unless the user explicitly asks.
- Do not tag which alias did each item when the aliases are one person.
- If excluding noisy results, add one short note after the list explaining the exclusion.
